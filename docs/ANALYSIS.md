# ADiGator-Embedded: Codebase Analysis — TODOs, Improvements & Testing

## Context

ADiGator v1.5 ("embedded edition") is a MATLAB source-transformation automatic differentiation tool. The v1.5 additions (by Pedro Lourenço @ GMV, 2025) introduce three embedding modes (`classic`, `coderload`, `inline`) so that generated derivative files can be compiled to C via MATLAB Coder/Embedded Coder without runtime global variables or `.mat` file loads.

This document audits the embedding subsystem (`embedding/`) and the broader library, identifying bugs, quality issues, and testing gaps prioritised by impact.

---

## 1. Explicit TODOs in the Code

**`embedding/adigatorGenDerFile_embedded.m:53–58`**

```matlab
%   TODO:
%       - Remove the computation of unnecessary variables by running CHECKCODE
%         on each file. Some functions load the mat file and assign Gator*Data
%         structures but don't use them. Each ADiGator derivative function
%         returns the desired derivative as well as all lower order derivatives
%         and the function itself. This might not be desirable in embedded code,
%         where only the minimum number of computations should take place.
```

In embedded modes the generated file computes Jacobian + function value + (if Hessian) gradient + function, even when the caller only needs the top-level derivative. For size- and performance-constrained embedded targets, dead code elimination is important. The TODO suggests using MATLAB's `checkcode`/`mlint` to detect unused assignments and then stripping them from the generated file.

## 2. Bugs (ordered by severity)

### BUG-1 · Variable name typo — potential runtime crash
**File:** `lib/@cada/adigatorAnalyzeForData.m` (~line 62)
**Severity:** High

```matlab
if size(ForLengths,2) < OuterLoopMaxLength
    ForLengths(end, OuterLoopMaxLenght) = 0;   % <-- "Lenght" misspelled
end
```

`OuterLoopMaxLenght` is undefined; `OuterLoopMaxLength` is the correct variable. MATLAB throws
`"Undefined variable 'OuterLoopMaxLenght'"` whenever the condition is true. This only fires in
specific nested-loop scenarios, so it may go unnoticed until triggered by a user with a
multi-iteration nested loop.

**Fix:** Rename `OuterLoopMaxLenght` → `OuterLoopMaxLength` on that single line.

---

### BUG-2 · `fprintf` format-string misuse produces incorrect generated file header
**File:** `embedding/structure_to_embed_mfile.m:38`
**Severity:** Medium

```matlab
fprintf(fid, '%% Auto-generated by structure_to_embed_mfile on %s\n% Helper file for ADiGator generated derivatives', ts);
```

The literal `% Helper file...` inside the format string contains an unescaped `%`. MATLAB's
`fprintf` interprets it as a format specifier, producing a warning or malformed output in the
generated file header.

**Fix:** Replace `'% Helper file'` with `'%% Helper file'` in the format string.

---

### BUG-3 · `coder.load` path not qualified in `coderload` mode
**File:** `embedding/adigator_patch_derivative.m:89`
**Severity:** Medium

```matlab
loading_call = "if isempty(" + globalName + "); " + globalName + ...
    " = coder.load('" + deriv_filename + ".mat','" + subfun_list{fun} + "'); end";
```

`deriv_filename` is the bare function name (e.g. `myfun_Jac`). If the user set `opts.path` to a
non-current directory, the `.mat` file will be placed there, but the generated `coder.load` call
will not include that path. At code-generation time the file will be unfindable.

**Fix:** Pass the full, qualified `.mat` path (`AdigatorGeneratedFiles(derf).mat`, already
available in the caller) into `adigator_patch_derivative` and use it when constructing the
`coder.load` string.

---

### BUG-4 · Integer array types silently lose type in `structure_to_embed_mfile`
**File:** `embedding/structure_to_embed_mfile.m`
**Severity:** Low–Medium

`emit_numeric` uses `pick_fmt` which only distinguishes `single` vs `double`. Integer types
(`int8`, `int16`, `int32`, `int64`, `uint8`, etc.) fall through without a type-cast wrapper.
The generated literal will be a plain `double`, silently changing the stored type.

**Fix:** In `emit_numeric`, detect integer classes with `isinteger(A)` and emit
`int32([...])` / `uint32([...])` etc. wrappers around the literal vector.

## 3. Code Quality Improvements

### Q-1 · Dual side-effect in `adigator_patch_derivative`
**File:** `embedding/adigator_patch_derivative.m:118–120`

The function both *writes the patched text back to the original file* **and** *returns the text as
`txt`*. Call sites in `adigatorGenDerFile_embedded.m` use the return value to later append content
into the wrapper; the in-place file write (line 119) is therefore a redundant side-effect on a
file that is immediately deleted (lines 140, 153).

**Improvement:** Separate concerns — either always return text and let the caller write, or always
write in-place and return nothing. The current dual behaviour makes control flow hard to follow and
creates an implicit ordering dependency.

---

### Q-2 · Magic number `8` for `'ADiGator'` prefix detection
**File:** `lib/adigatorFunctionInitialize.m:296` (and similar sites)

```matlab
if length(FunctionStr) > 8
```

The number `8` is the length of the string `'ADiGator'` but appears with no documentation.
Introduce a named constant `ADIGATOR_PREFIX = 'ADiGator'` and replace magic comparisons with
`length(ADIGATOR_PREFIX)`.

---

### Q-3 · Fragile proximity heuristic in `find_in_file`
**File:** `embedding/adigator_patch_derivative.m:150–155`

The `once > 1` mode stops collecting matches when two consecutive hits are more than 2 lines
apart. This heuristic breaks if a code block is reformatted or if line spacing changes between
MATLAB versions.

**Improvement:** For the specific use case (finding consecutive `GatorXData` assignments), iterate
over known Gator variable names derived from the `.mat` structure rather than relying on line-gap
proximity.

---

### Q-4 · Downcasting threshold may incorrectly cast near-integer floats
**File:** `embedding/adigatorGenDerFile_embedded.m:209`

```matlab
if all(abs(A(:) - round(A(:))) < 1e-12)
    A = uint32(A);   % or int32
end
```

Near-integer floating-point values (e.g. `1.0 + 2*eps`) that are semantically non-integer could
be incorrectly cast. Consider tightening the threshold to `eps(max(abs(A(:))))` or only
downcasting arrays that were stored as integers in the original `.mat`.

---

### Q-5 · Extensive global state hinders testability
**Files:** `adigator.m:70`, `lib/@cada/*.m`, `lib/adigatorFunctionInitialize.m`, etc.

Four global structures (`ADIGATOR`, `ADIGATORFORDATA`, `ADIGATORDATA`,
`ADIGATORVARIABLESTORAGE`) are declared across ~30 files. This makes it impossible to run
isolated unit tests or parallel differentiation without state leakage. The embedded modes address
the *generated* code side, but the *generation* side still relies heavily on globals.

**Short-term improvement:** Explicitly clear and re-initialise globals at the start and end of
`adigator.m` so that successive test runs do not inherit stale state.
**Long-term:** Encapsulate global state into a context object (`AdigatorContext`) passed through
the call chain.

---

### Q-6 · `updatestruct` type coercion silently truncates values
**File:** `embedding/updatestruct.m:40`

```matlab
out.(input_fields{ii}) = feval(class(out.(input_fields{ii})), in.(input_fields{ii}));
```

If the default field is `uint8` and the user passes a `double` that overflows, the value is
silently clipped. No warning is issued.

**Fix:** Add a check: if the cast changes the value (`~isequal(cast, original)`), issue a
`warning`.

---

### Q-7 · `adigatorAnalyzeForData.m` is a 2,703-line monolith
**File:** `lib/@cada/adigatorAnalyzeForData.m`

The file contains the primary loop analysis function plus ~10 large nested sub-functions. Each
sub-function (`GetDataDependencies`, `RemoveUnnneededIndices`, `GetSubsrefInds`, etc.) is over
100 lines with sparse internal documentation. Note also the typo in `RemoveUnnneededIndices`
(triple `n`).

**Improvement:** Extract nested sub-functions into separate private files in `@cada/private/` to
improve navigability and testability.

## 4. Unit Test Gaps & Recommended Test Suite

### Current state
| Item | Status |
|---|---|
| Unary math derivative rules | ✅ `unit_tests/test_unarymath_rules.m` |
| Binary math derivative rules | ❌ missing |
| Embedding subsystem | ❌ missing |
| Jacobian/Hessian generation utilities | ❌ missing |
| Loop analysis | ❌ missing |
| CI/CD automation | ❌ missing |

### Issues in the existing test (`test_unarymath_rules.m`)

1. **Forward finite differences** `(f(x+h)−f(x))/h` are O(h) accurate. Central differences
   `(f(x+h)−f(x−h))/(2h)` are O(h²) and catch more subtle rule errors.
2. `addpath('tmp')` is never undone (`rmpath`), polluting the MATLAB path for subsequent tests.
3. No formal assertions — only `fprintf` output. Should return a non-zero value or `error()` so
   CI can detect failure.
4. `movefile([dname,'.*'],'tmp/')` relies on glob expansion in `movefile`, which is not
   documented and may not be portable across OS/MATLAB versions.
5. No teardown — the `tmp/` directory and generated `.m` files are left on disk.

### Recommended new unit test files

| File | What it verifies |
|---|---|
| `unit_tests/test_structure_to_embed_mfile.m` | Scalar, vector, matrix, nested struct, cell array, char, integer-type inputs; verify generated `.m` is syntactically valid; verify complex literal round-trip |
| `unit_tests/test_prune_adigator_mat.m` | `prune_adigator_mat` keeps `Index*` fields, removes non-`Index` non-empty fields, correctly downcasts integer-valued doubles; handles missing function names gracefully |
| `unit_tests/test_adigator_patch_derivative.m` | Inline mode: `global` removed, `coder.const(data_func())` inserted, `%#codegen` present; coderload mode: `global→persistent`, `coder.load(...)` inserted; verify `ADiGator_LoadData` subfunction removed |
| `unit_tests/test_updatestruct.m` | Nested struct merge, field addition with `addfields=true`, type coercion passthrough |
| `unit_tests/test_unarymath_rules.m` *(improved)* | Switch to central differences; add `rmpath`/cleanup teardown; `assert(violations==0)` |

## 5. Integration & System Testing Strategy

### Integration test: full embedded pipeline

**File:** `unit_tests/test_embedded_pipeline.m`

Algorithm:
1. Define a simple known function, e.g. `f(x) = 0.5 * sum(x.^2)`, Jacobian = `x'`.
2. For each embedding mode (`'c'`, `'l'`, `'i'`):
   a. Call `adigatorGenDerFile_embedded('jacobian', 'test_quad', {ax}, opts)`
   b. Verify the expected output files exist (count and naming per mode).
   c. Verify absence of `global` / `load` statements in the generated `.m` for modes `'l'` and
      `'i'`.
   d. Verify `%#codegen` is present in modes `'l'` and `'i'`.
   e. Execute the generated wrapper with a known input and compare numerical output to the
      analytical Jacobian within tolerance `1e-10`.
3. Clean up all generated files after each sub-test.

**Regression functions** (use existing examples non-interactively):
- Brachistochrone Jacobian — loop-intensive
- Brusselator ODE Jacobian — stiff nested loops
- Logsumexp Hessian — second-order derivative

### System test: MATLAB Coder compilation

**File:** `unit_tests/test_coder_compile.m` *(requires Coder licence)*

1. Run the inline pipeline on a representative function.
2. Call `codegen` with appropriate `-config:mex` settings on the generated wrapper.
3. Assert no compilation errors.
4. Compare the MEX output to the MATLAB output numerically (tolerance `1e-10`).

### CI/CD

Add `.github/workflows/tests.yml` running MATLAB in `-batch` mode:

```yaml
- name: Run unit tests
  run: |
    matlab -batch "startupadigator; assert(test_unarymath_rules()==0)"
    matlab -batch "startupadigator; test_embedded_pipeline()"
    matlab -batch "startupadigator; test_structure_to_embed_mfile()"
    matlab -batch "startupadigator; test_adigator_patch_derivative()"
```

Make `examples/runAllExamples.m` accept a `--batch` flag argument to skip interactive prompts
so the full example suite can also be run in CI.
